6. Relaciones y claves foráneas

En una base de datos relacional, el valor está en conectar entidades. Las claves foráneas (FK) preservan la integridad referencial, garantizando que los datos relacionados existan y se mantengan coherentes. En esta sección diseñarás relaciones 1:N y N:M, crearás claves foráneas con acciones ON DELETE/UPDATE y practicarás consultas con JOIN (INNER, LEFT, RIGHT, FULL*).

*MySQL no implementa FULL OUTER JOIN de forma nativa; se simula con unión de LEFT y RIGHT.

Objetivos del bloque

  • Diferenciar relaciones 1:N y N:M (mediante tabla puente).
  • Definir y justificar claves foráneas con acciones ON DELETE/ON UPDATE.
  • Realizar consultas con JOIN para recuperar información combinada.
  • Aplicar todo lo anterior en un mini-proyecto “tienda_online”.

6.1 Diseño relacional: 1:N y N:M

Relación 1:N

Una fila de la tabla padre puede estar asociada con muchas filas de la tabla hija. Ejemplo: clientes (1) → pedidos (N). La FK se coloca en la tabla hija.

CREATE TABLE clientes (
  id INT AUTO_INCREMENT PRIMARY KEY,
  nombre VARCHAR(60) NOT NULL,
  email VARCHAR(120) UNIQUE
);

CREATE TABLE pedidos (
  id INT AUTO_INCREMENT PRIMARY KEY,
  fecha DATE NOT NULL,
  cliente_id INT NOT NULL,
  CONSTRAINT fk_pedidos_clientes
    FOREIGN KEY (cliente_id) REFERENCES clientes(id)
);

Relación N:M (tabla puente)

Para N:M se introduce una tabla intermedia que contiene al menos las FKs de ambas tablas. Ejemplo: productos ↔ categorias mediante producto_categoria.

CREATE TABLE productos (
  id INT AUTO_INCREMENT PRIMARY KEY,
  nombre VARCHAR(80) NOT NULL,
  precio DECIMAL(10,2) NOT NULL
);

CREATE TABLE categorias (
  id INT AUTO_INCREMENT PRIMARY KEY,
  nombre VARCHAR(60) NOT NULL UNIQUE
);

CREATE TABLE producto_categoria (
  producto_id INT NOT NULL,
  categoria_id INT NOT NULL,
  PRIMARY KEY (producto_id, categoria_id),
  CONSTRAINT fk_pc_prod FOREIGN KEY (producto_id) REFERENCES productos(id),
  CONSTRAINT fk_pc_cat  FOREIGN KEY (categoria_id) REFERENCES categorias(id)
);

6.2 Claves foráneas y acciones ON DELETE / ON UPDATE

Las FKs pueden definir qué hacer cuando cambia o se borra el registro referenciado.

  • RESTRICT: impide borrar/actualizar si hay referencias (por defecto en muchos motores).
  • CASCADE: propaga el borrado/actualización a las filas hijas.
  • SET NULL: coloca NULL en la FK de las filas hijas.
  • NO ACTION: equivalente a RESTRICT en MySQL.
ALTER TABLE pedidos
  ADD CONSTRAINT fk_pedidos_clientes
  FOREIGN KEY (cliente_id) REFERENCES clientes(id)
  ON DELETE RESTRICT
  ON UPDATE CASCADE;
Buenas prácticas:
  • Usa CASCADE con cautela: puede borrar grandes cascadas sin darte cuenta.
  • SET NULL requiere que la columna FK permita NULL.
  • Asegúrate de que las tablas usen un motor que soporte FKs (p. ej., InnoDB).
  • Indexa siempre las columnas de FK para mejorar el rendimiento de los JOIN.

6.3 JOINs: INNER, LEFT, RIGHT y (simulación) FULL

INNER JOIN

Devuelve solo las filas con coincidencia en ambas tablas.

SELECT p.id, p.fecha, c.nombre
FROM pedidos p
INNER JOIN clientes c ON c.id = p.cliente_id;

LEFT JOIN

Devuelve todas las filas de la tabla izquierda y, cuando no hay coincidencia, valores NULL a la derecha.

-- clientes sin pedidos (detectamos NULL en el lado derecho)
SELECT c.id, c.nombre, p.id AS id_pedido
FROM clientes c
LEFT JOIN pedidos p ON p.cliente_id = c.id
WHERE p.id IS NULL;

RIGHT JOIN

Simétrico de LEFT JOIN (suele ser menos usado si controlas el orden de las tablas).

FULL OUTER JOIN (simulación)

-- Unión de LEFT y RIGHT excluyendo duplicados
SELECT c.id, c.nombre, p.id AS id_pedido
FROM clientes c
LEFT JOIN pedidos p ON p.cliente_id = c.id
UNION
SELECT c.id, c.nombre, p.id AS id_pedido
FROM clientes c
RIGHT JOIN pedidos p ON p.cliente_id = c.id;
Tip: Prefiere JOIN con alias de tabla (c, p) y nombra columnas en SELECT para legibilidad.

6.4 Mini-proyecto guiado: “tienda_online”

Crea el esquema y practica relaciones, FKs y JOINs.

Esquema

CREATE DATABASE tienda_online CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci;
USE tienda_online;

CREATE TABLE usuarios (
  id INT AUTO_INCREMENT PRIMARY KEY,
  nombre VARCHAR(80) NOT NULL,
  email VARCHAR(120) NOT NULL UNIQUE,
  fecha_registro DATE NOT NULL
);

CREATE TABLE productos (
  id INT AUTO_INCREMENT PRIMARY KEY,
  nombre VARCHAR(100) NOT NULL,
  precio DECIMAL(10,2) NOT NULL,
  stock INT NOT NULL DEFAULT 0
);

CREATE TABLE pedidos (
  id INT AUTO_INCREMENT PRIMARY KEY,
  usuario_id INT NOT NULL,
  fecha DATETIME NOT NULL,
  CONSTRAINT fk_pedidos_usuarios
    FOREIGN KEY (usuario_id) REFERENCES usuarios(id)
    ON DELETE RESTRICT ON UPDATE CASCADE
);

CREATE TABLE lineas_pedido (
  pedido_id INT NOT NULL,
  producto_id INT NOT NULL,
  cantidad INT NOT NULL CHECK (cantidad > 0),
  precio_unit DECIMAL(10,2) NOT NULL,
  PRIMARY KEY (pedido_id, producto_id),
  CONSTRAINT fk_lp_ped   FOREIGN KEY (pedido_id)  REFERENCES pedidos(id)   ON DELETE CASCADE,
  CONSTRAINT fk_lp_prod  FOREIGN KEY (producto_id) REFERENCES productos(id) ON DELETE RESTRICT
);

Datos de ejemplo

INSERT INTO usuarios (nombre, email, fecha_registro) VALUES
('Ana', 'ana@example.com', '2024-09-01'),
('Luis','luis@example.com','2024-09-10'),
('Sara','sara@example.com','2024-09-15');

INSERT INTO productos (nombre, precio, stock) VALUES
('Teclado', 49.90, 20), ('Ratón', 19.90, 50), ('Monitor 24"', 149.00, 10);

INSERT INTO pedidos (usuario_id, fecha) VALUES
(1, NOW()), (1, NOW()), (2, NOW());

INSERT INTO lineas_pedido (pedido_id, producto_id, cantidad, precio_unit) VALUES
(1, 1, 1, 49.90), (1, 2, 2, 19.90),
(2, 3, 1, 149.00),
(3, 2, 1, 19.90);

Consultas clave (JOIN + agregación)

-- a) Pedidos con nombre de usuario
SELECT p.id, u.nombre, p.fecha
FROM pedidos p
JOIN usuarios u ON u.id = p.usuario_id
ORDER BY p.id;

-- b) Importe por pedido
SELECT lp.pedido_id,
       SUM(lp.cantidad * lp.precio_unit) AS total_pedido
FROM lineas_pedido lp
GROUP BY lp.pedido_id
ORDER BY lp.pedido_id;

-- c) Top productos por ventas (unidades)
SELECT pr.nombre, SUM(lp.cantidad) AS unidades
FROM lineas_pedido lp
JOIN productos pr ON pr.id = lp.producto_id
GROUP BY pr.id, pr.nombre
ORDER BY unidades DESC;

-- d) Usuarios sin pedidos (LEFT JOIN + NULL)
SELECT u.id, u.nombre
FROM usuarios u
LEFT JOIN pedidos p ON p.usuario_id = u.id
WHERE p.id IS NULL;
Extensión opcional: añade tabla categorias y producto_categoria (N:M) y consulta: “productos con sus categorías” y “categorías sin productos”.

6.5 Actividades y evaluación

  1. Modelado: Dibuja (en papel o herramienta) el diagrama E/R de “tienda_online” con cardinalidades y claves.
  2. Integridad: Cambia la acción de ON DELETE en lineas_pedido y justifica el impacto.
  3. Consultas: Obtén (i) pedidos con total > 100 €, (ii) productos sin vender, (iii) ingresos por día/mes.
  4. Rendimiento: Crea índices en columnas FK y compara tiempos en consultas con y sin índice.

Autoevaluación (escala 1–5)

  • Distingo correctamente 1:N y N:M y sé cuándo usar tabla puente.
  • Defino FKs y acciones ON DELETE/UPDATE adecuadas al caso de uso.
  • Escribo consultas INNER/LEFT JOIN con soltura y comprendo sus resultados.
  • Calculo totales por pedido y rankings de ventas con agregaciones.
