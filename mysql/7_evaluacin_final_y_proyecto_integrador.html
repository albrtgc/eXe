7. Evaluación final y proyecto integrador

Esta sección verifica el logro de los objetivos del REA mediante un cuestionario sumativo, un proyecto práctico integrador y una autoevaluación. El alumnado aplicará instalación, modelado relacional, integridad referencial y consultas SQL (DML, JOINs, agregaciones y subconsultas).

Objetivos del bloque

  • Comprobar el dominio de MySQL en un caso práctico realista.
  • Diseñar e implementar un esquema relacional con integridad y datos de ejemplo.
  • Redactar consultas SQL que resuelvan necesidades típicas de explotación.
  • Reflexionar sobre el propio proceso de aprendizaje y áreas de mejora.

7.1 Cuestionario sumativo (ejemplos)

Sugerencia eXe: crea estas preguntas en iDevices de tipo test/verdadero-falso o importa GIFT a Moodle.

  1. Selección de BD: ¿Qué instrucción establece la BD activa?
    a) OPEN DB b) SET DB c) USE d) SELECT DATABASE
  2. Integridad: La clave foránea garantiza…
    a) Rendimiento b) Integridad referencial c) Redundancia d) Copia de seguridad
  3. Agregación: Media de precio por categoría: 
    SELECT categoria, AVG(precio) FROM productos GROUP BY categoria;
  4. JOIN: Señala la opción correcta para obtener pedidos con nombre del cliente: 
    SELECT p.id, c.nombre FROM pedidos p JOIN clientes c ON c.id = p.cliente_id;
  5. Subconsulta: Productos con precio mayor que la media: 
    SELECT nombre FROM productos WHERE precio > (SELECT AVG(precio) FROM productos);

7.2 Proyecto práctico integrador: tienda_curso

Implementa una base de datos de comercio electrónico simplificada. Entrega scripts SQL y evidencias.

7.2.1 Requisitos de diseño

  • BD: tienda_curso (charset utf8mb4, colación recomendada utf8mb4_general_ci o utf8mb4_0900_ai_ci si procede).
  • Tablas mínimas:
    • usuarios(id PK, nombre, email UNIQUE, fecha_registro DATE)
    • productos(id PK, nombre, precio DECIMAL, stock INT)
    • pedidos(id PK, usuario_id FK, fecha DATETIME)
    • lineas_pedido(pedido_id FK, producto_id FK, cantidad > 0, precio_unit DECIMAL, PK compuesta)
    • Opcional +1: categorias y producto_categoria (N:M)
  • Restricciones: PK, NOT NULL, CHECK(cantidad > 0), FK con acciones justificadas (RESTRICT, CASCADE, SET NULL).
  • Índices: en todas las FKs y en columnas de búsqueda frecuente (p. ej., email).

7.2.2 Datos de ejemplo

Inserta un mínimo de: 5 usuarios, 10 productos, 8 pedidos y sus líneas. Debe haber al menos 1 usuario sin pedidos y 1 producto sin vender.

7.2.3 Consultas obligatorias

  1. Pedidos con cliente: id pedido, nombre cliente y fecha (JOIN).
  2. Importe por pedido: suma de cantidad * precio_unit (GROUP BY).
  3. Top ventas: productos ordenados por unidades vendidas (JOIN + SUM + ORDER).
  4. Clientes sin pedidos: LEFT JOIN + IS NULL.
  5. Productos sobre la media: precio > media global (subconsulta).
  6. Ingresos por mes: total de ventas agrupado por DATE_FORMAT(fecha, '%Y-%m').

7.2.4 Entregables

  • 01_creacion_esquema.sql (CREATE DATABASE/TABLE + restricciones + índices)
  • 02_datos_demo.sql (INSERTs)
  • 03_consultas_obligatorias.sql (comentadas)
  • Capturas: evidencia de estructura, FKs y resultados de consultas.
  • README.md: decisiones de diseño y justificación de ON DELETE/UPDATE.

7.4 Autoevaluación (escala 1–5)

  • He creado un esquema en 3FN con restricciones adecuadas.
  • Comprendo y justifico mis elecciones de ON DELETE/UPDATE.
  • Domino SELECT con JOIN y agregaciones para resolver métricas.
  • Uso subconsultas cuando simplifican la solución.
  • Mantengo scripts ordenados y reproducibles.

7.5 Criterios de calificación y entrega

  • Puntuación: Test (20%) + Proyecto (70%) + Autoevaluación (10%).
  • Formato de entrega: ZIP con 01_creacion_esquema.sql, 02_datos_demo.sql, 03_consultas_obligatorias.sql, capturas y README.
  • Entrega: en Moodle → Tarea “Proyecto integrador MySQL”.
  • Originalidad: se penaliza el plagio. Justifica tus decisiones en el README.

Consejo: Prueba los scripts en un entorno limpio (BD vacía) antes de entregar. Incluye DROP DATABASE IF EXISTS al inicio para facilitar la corrección.
